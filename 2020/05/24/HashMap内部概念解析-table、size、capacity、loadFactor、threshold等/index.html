HashMap内部概念解析-table、size、capacity、loadFactor、threshold等

发表于 2020-05-24

HashMap是Java中最常用的集合类之一，内部实现原理对大部分人来说也是比较熟悉的，本文通过剖析HashMap内部的各个概念来深入理解HashMap的实现原理与一些细节。若无特别说明，以下内容描述的都是1.8版本中HashMap

table

众所周知HashMap内部存储核心数据的是一个数组，称之为table。数组中每一个元素称之为桶（bucket）。每个桶内都可以装0个、1个或多个Node，而Node是Map.Entry的实现类，我们可以理解成一个Node即为一个key-value对。那么一个桶内是如何盛装多个Node的呢？有以下两种方式：

  • 链表：多个Node组织为一个链表的形式，每个桶只需要保存链表的头节点，也就相当于保存了整个链表了
  • 红黑树：多个Node组织为一棵红黑树，每个桶只需要保存红黑树的根节点，也就相当于保存了整棵红黑树了
    最开始时，所有桶都是用链表的方式盛装多个Node的。但当某个桶内元素过多后，为了提升查询效率，这个桶内的链表会转为红黑树；同样，当某个桶内元素过少时，这个桶内的红黑树又会转化为链表。

size

size是整个HashMap中所有Node的总数，也就是所有桶内Node数的总和。通过调用size()方法可以获取到当前HashMap的size，但方法内部并不是每一次调用都去计算每个桶内Node数然后求和返回，实际上HashMap中有一个成员变量size保存当前的Node总数，size()方法仅仅是将其返回而已。size初始值为0，在put(K, V)方法中put完成后会对size进行加1操作，同样在remove(Object, Object)方法中remove完成后会对size进行减1操作，这样size就始终为HashMap当前的Node总数。

capacity

capacity中文为容量，乍一看会以为是用于限制HashMap中Node个数的，然而并不是。如果不考虑int溢出或内存溢出的话，HashMap对于Node个数理论上是没有限制的。capacity实际上是指table数组的长度，也就是桶的个数。同样，我们常听到的HashMap的“扩容”也并不是提升Node个数的上限，而是增加table数组的长度，允许更多桶存在。

HashMap负责扩容的方法是resize()，它也负责初始化table数组。每次扩容，resize()方法都会创建一个新数组newTab，长度为旧数组oldTab的两倍，将table指向newTab，然后将oldTab中所有桶的所有Node rehash到newTab，至此扩容完毕。

MAXIMUM_CAPACITY
